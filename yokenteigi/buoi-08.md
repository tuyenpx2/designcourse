## Slide 4: よくある失敗パターン — 実務から学ぶ

### 失敗 Case 1: スコープの曖昧さ

**状況:**
> 要件定義書に「ユーザー管理機能」と書いた。
> 開発終盤に「ユーザーの一括インポート機能はないのか」とクレーム。

**原因:**
- 「ユーザー管理」の範囲を明示していなかった
- 機能一覧が「登録・更新・削除」だけで、インポート/エクスポートを記載していなかった

**対策:**
```
✅ 機能一覧は「対象外」も明示する
例:
- ✅ 対象: 個別登録、個別編集、削除
- ❌ 対象外: CSVインポート（次フェーズ検討）
```

---

### 失敗 Case 2: 非機能要件なし = 後から揉める

**状況:**
> 要件定義書に性能要件を書かなかった。
> リリース後、同時接続50人で画面が10秒かかると苦情。
> 客: 「遅い」、開発: 「仕様に書いていない」

**原因:**
- 非機能要件を「あとで考えよう」と先送り
- 「常識的に速いもの」という暗黙の了解

**対策:**
```
✅ 必ずプロジェクト初期に非機能要件をヒアリング
✅ 「未定」でも一旦仮の数値を入れ、後で修正
✅ 仮の数値でも「仮」と明記して承認を取る
```

---

### 失敗 Case 3: 例外フローの軽視

**状況:**
> 決済機能で「支払い完了」のフローしか記載していなかった。
> 実装後、「支払い処理中にブラウザを閉じたら二重課金になる」バグが発覚。

**原因:**
- 例外フロー（ネットワーク切断、ブラウザバック等）を考慮しなかった
- 設計段階で発見できず、実装・テスト段階で発覚

**対策:**
```
✅ 全ての外部システム連携には必ず例外フローを書く
✅ 「What if ○○が失敗したら？」を全ステップで問う
✅ 特に決済・メール・外部APIは念入りに
```

---

### 失敗 Case 4: 議事録と要件の乖離

**状況:**
> ヒアリング時に「承認は2段階でいい」と言っていたのに、
> 要件定義書に「3段階承認」と書いてしまった。
> リリース後に発覚、改修コストが発生。

**原因:**
- 議事録を送らず、メモが曖昧なまま要件定義書を書いた
- 決定事項の確認が取れていなかった

**対策:**
```
✅ 議事録は24時間以内に送付、「内容に相違なければご確認ください」
✅ 重要な決定事項は必ず「〜という認識でよろしいですか？」で再確認
✅ 要件定義書のレビュー時に議事録と照合する
```

## Slide 5: 実務でのワークフロー — 現場のリアル

### 典型的なプロジェクトの流れ

```
Week 1-2: 킥오프・初回ヒアリング
    │ 議事録、初回質問書を送付
    │
Week 3-4: 業務フロー作成
    │ As-Is流れ確認 → To-Be案作成 → 顧客レビュー
    │
Week 5-6: 機能一覧・機能詳細 (Draft)
    │ 優先度の高い機能から順に作成
    │
Week 7: 非機能要件・画面遷移図
    │ 顧客へ 質問書#2 送付
    │
Week 8-9: ドキュメントレビュー
    │ 顧客レビュー → 修正 → 再レビュー（2-3往復）
    │
Week 10: 承認・Sign-off
    │ 顧客の署名・捺印 or 電子承認
    │
Week 11: 基本設計フェーズへ
```

### Sign-offの重要性

> **要件定義書のSign-offなしに設計・開発を始めてはいけない**
>
> なぜなら:
> - 承認なし = いつでも「言っていない」と言われるリスク
> - 変更が発生した場合の基準がない
> - チームが「何を作るか」の合意がない

## Slide 6: バージョン管理と変更管理の実務

### 変更が発生したときの正しい対処

**NG対応:**
- こっそり修正してバージョンを変えない
- メールだけで変更を伝え、文書を更新しない

**OK対応:**
```
Step 1: 変更依頼を記録
   → 課題管理表 or 変更管理表に記載

Step 2: 変更の影響を分析
   → スコープ/スケジュール/コストへの影響を評価

Step 3: 変更内容を承認
   → 顧客とPMの両方から承認を取る

Step 4: ドキュメントを更新
   → 改訂履歴に記載、バージョンを上げる

Step 5: 関係者に配布
   → 全ステークホルダーに最新版を送付
```

### 変更管理表フォーマット

| 変更ID | 依頼日 | 依頼者 | 変更内容 | 影響範囲 | 対応方針 | 承認日 | 反映バージョン |
|--------|--------|--------|---------|---------|---------|--------|--------------|
| CR-001 | 3/15 | 田中様 | ポイント機能を追加 | F015新規追加、工数+5日 | Phase 1に含める | 3/20 | v2.0 |

## Slide 7: キャリアへの活かし方

### 要件定義スキルがあると...

**BA (Business Analyst) として:**
- 要件定義書の作成・管理が主業務
- 顧客との窓口役として価値が高い

**SE / PL として:**
- 要件定義を理解した開発ができる
- 「作る」だけでなく「何を作るか」に関与できる

**PM として:**
- スコープ管理の基礎として不可欠
- リスク管理、変更管理に直結

### 市場価値

```
要件定義できない SE   : ★★☆☆☆ (コーダー)
要件定義できる SE     : ★★★☆☆ (上流工程参加可)
要件定義が得意な SE   : ★★★★☆ (PL候補)
要件定義 + 顧客折衝   : ★★★★★ (PM/BA候補)
```

## Slide 8: 総まとめ — 要件定義書の黄金ルール

### 11の黄金ルール

```
①  スコープを最初に決める
    → 何を作らないかを明示する

②  全ての要件を定量化する
    → 「速い」ではなく「3秒以内」

③  例外フローを絶対に省略しない
    → Happy Pathは全体の20%

④  用語を定義して一貫して使う
    → 「ユーザー」か「会員」かを統一

⑤  議事録は24時間以内に送付
    → 言った言わない問題を防ぐ

⑥  機能要件はMECEに
    → 漏れなく・ダブりなく

⑦  全てにIDを振る
    → F001, S001, Q001... 追跡可能にする

⑧  要件の根拠を残す
    → なぜこの要件が必要かをコメントに

⑨  Sign-offなしに開発を始めない
    → 承認 = プロジェクトの契約

⑩  バージョン管理を厳密に
    → 改訂履歴に日付・変更者・内容

⑪  課題管理表を最後まで更新する
    → 「未決のまま」でゴー・ライブしない
```
