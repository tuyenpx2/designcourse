## Slide 2: Case Study — Hệ thống Quản lý & Mượn thiết bị văn phòng

### Thông tin khách hàng

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
顧客:  テクノオフィス株式会社
業種:  IT サービス企業
規模:  社員250名、オフィス3拠点（東京・大阪・福岡）
管理対象機器数: 約500台
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Yêu cầu thô từ khách hàng (原文)

> 「今は社内の設備やPCの管理をExcelでやっているんですが、誰が何を借りているかわからなくなってしまっていて困っています。
>
> 社員がPCや会議室の備品、社用車、モバイルWiFiなどを借りる時に、今は総務部に電話かメールで連絡しているんですが、管理が追いつかなくて。
>
> システム化して、社員が自分でWeb上から貸出申請できるようにしたいんです。申請があったら総務担当者が確認して貸し出す、という流れにしたいです。
>
> 各機器が今どういう状態なのか（誰かが使っているのか、空いているのか）もリアルタイムで見えるようにしてほしいです。
>
> あと、機器の追加・廃棄・修理対応なんかも管理できれば助かります。返却期限を過ぎた場合に自動で通知が来るといいですね。
>
> セキュリティも気になっていて、誰でもどの機器でも申請できるのはまずいかと思っています。」

## Slide 11: 機能詳細 — F030 貸出申請 (みんなで書く)

```
機能ID:  F030
機能名:  貸出申請
優先度:  必須
対象:    ログイン済みUser / Admin

【前提条件】
- ユーザーがログイン済みであること
- 対象機器が「利用可能」ステータスであること
- ユーザーが同カテゴリの機器を既定数以上借りていないこと
  （例: PCは同時に1台まで）

【入力項目】
| フィールド | 必須 | 形式 | バリデーション |
|-----------|------|------|--------------|
| 機器ID | ✓ | — | 利用可能ステータスの機器のみ |
| 貸出開始日 | ✓ | YYYY/MM/DD | 本日以降 |
| 返却予定日 | ✓ | YYYY/MM/DD | 開始日より後、カテゴリ上限以内 |
| 利用目的 | ✓ | テキスト | 最大200文字 |

【基本フロー】
1. ユーザーが機器一覧 or 機器詳細から「申請する」をクリック
2. 申請フォームが表示される（機器情報は自動入力）
3. ユーザーが貸出期間・利用目的を入力
4. ユーザーが「申請」ボタンをクリック
5. システムがバリデーションを実行
6. システムが申請レコードを「申請中」ステータスで登録
7. システムが機器ステータスを「予約済み」に変更
8. システムがAdminへ申請通知メールを送信（F050）
9. 申請完了画面を表示、申請番号を通知

【代替フロー】
5a. 同カテゴリ機器の貸出上限を超過している場合
→ 「PCはすでに1台貸出中です。返却後に申請してください」
→ 申請不可

【例外フロー】
5b. 申請処理中に機器ステータスが変化した場合（楽観ロック競合）
→ 「この機器は現在申請できません。一覧に戻って再確認してください」
→ 申請レコードは登録しない、機器ステータスは変更しない

6a. DB保存エラー
→ トランザクションロールバック
→ 機器ステータスは変更しない
→ エラーメッセージ[ERR-APPLY-001]を表示

8a. 通知メール送信失敗
→ 申請レコード・機器ステータスは維持
→ 15分後に1回自動再送
→ 画面に「管理者への通知に失敗しました。担当者に直接ご連絡ください」
```

## Slide 13: まとめと宿題

### まとめ
- 権限設計は最初に明確化する（機能一覧の前に確定させる）
