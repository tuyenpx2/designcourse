## Slide 2: Case Study — AI請求書自動処理システム

### Thông tin khách hàng

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
顧客:   アカウントプロ株式会社
業種:   中堅会計事務所（会計・税務・記帳代行サービス）
規模:   経理担当スタッフ30名（ベトナム人スタッフ中心）
クライアント数: 約200社（中小企業の記帳代行）
処理量: 月間約8,000枚の請求書・領収書・納品書
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 顧客からの生の要望（原文）

> 「うちのスタッフは毎日4〜6時間、紙やPDFの請求書を見ながら会計ソフトにデータを手入力しているんです。もう限界で。
>
> 取引先からFAX・メール・郵便でいろんな書類が届くんですが、電気代の領収書、仕入れの請求書、クレジットカード明細、宅配の領収書など種類もバラバラで。
>
> 読み取って自動で入力してくれるシステムがほしいです。スキャンしたりスマホで撮影したりして、それをアップロードするだけで仕訳が自動で出てきてほしい。
>
> ただ、AIが間違える可能性もあると思うので、スタッフが最終チェックできるようにしてほしいです。自動で全部やるんじゃなくて、「AIが提案して人間が確認」という流れがいいです。
>
> 1日に何百枚も処理することもあるので、まとめてアップロードできると助かります。処理が終わったら通知してほしいです。
>
> 今は弥生会計を使っていますが、データをそのまま取り込める形式で出力できれば嬉しいです。」

### Yêu cầu quan trọng mà khách hàng không nói rõ

| Yêu cầu ẩn | Lý do suy ra | Mức độ |
|-----------|-------------|--------|

| Retry khi AI xử lý thất bại | Mạng lỗi, server AI lỗi → cần cơ chế thử lại | 🔴 Critical |

| Ngưỡng confidence AI (ví dụ: <80% → NEEDS_REVIEW) | Khách hàng nói "AI có thể sai" | 🟡 High |
| Master table: nhà cung cấp ↔ mã tài khoản | Smart Mapping cần bảng mapping | 🟡 High |
| Audit log đầy đủ — ai sửa gì, lúc nào | Dữ liệu tài chính — yêu cầu pháp lý | 🟡 High |
| Export CSV theo định dạng 弥生会計 | Khách hàng đang dùng 弥生会計 | 🟡 High |
| Tìm kiếm, lọc, phân trang ≥100k records | 8,000 hóa đơn/tháng × 12 tháng | 🟡 High |
| Báo cáo accuracy của AI theo thời gian | Cần đo hiệu quả hệ thống | 🟢 Medium |
| Camera capture trên mobile | "スマホで撮影" → cần Flutter mobile | 🟡 High |

### ⚠️ Điểm quan trọng nhất của buổi học

```
❗ Junior engineer HAY MẮC LỖI:
   Không định nghĩa đầy đủ STATE TRANSITIONS cho invoice

❌ Sai:
   Status: PROCESSING → COMPLETED

✅ Đúng — phải định nghĩa TẤT CẢ các trạng thái:
   UPLOADED → QUEUED → PROCESSING → COMPLETED
                                  → FAILED
                                  → NEEDS_REVIEW

   Và phải định nghĩa:
   - Ai có quyền trigger mỗi transition?
   - Điều kiện để chuyển sang mỗi trạng thái?
   - Khi FAILED → có retry không? Bao nhiêu lần?
   - NEEDS_REVIEW → user approve → COMPLETED
   - NEEDS_REVIEW → user reject → FAILED
```

### 請求書ステータス遷移図 — 最重要

```
                    ┌─────────────────────────────────┐
                    │         AI-IA ステータス遷移       │
                    └─────────────────────────────────┘

[1] ユーザーがファイルをアップロード
         │
         ▼
  ┌─────────────┐
  │  UPLOADED   │  ← S3保存完了、DBレコード作成
  └─────────────┘
         │  Laravelがキューにジョブをプッシュ
         ▼
  ┌─────────────┐
  │   QUEUED    │  ← Redisキューに積まれた状態
  └─────────────┘
         │  Workerがジョブを取得
         ▼
  ┌─────────────┐
  │ PROCESSING  │  ← Python FastAPI 処理中
  └─────────────┘
         │
    ┌────┴────────────────────────┐
    │                             │
    ▼                             ▼
┌──────────┐              ┌─────────────────┐
│COMPLETED │              │  NEEDS_REVIEW   │
│(信頼度   │              │  (信頼度 < 閾値 │
│≥ 閾値)   │              │   or 要確認項目)│
└──────────┘              └─────────────────┘
                                  │
                     ┌────────────┴────────────┐
                     │                          │
                [User承認]                  [User却下]
                     │                          │
                     ▼                          ▼
               ┌──────────┐             ┌──────────┐
               │COMPLETED │             │  FAILED  │
               └──────────┘             └──────────┘

    ※ PROCESSING → FAILED: AIサービスエラー、タイムアウト
       FAILED → QUEUED: リトライ（最大3回）
       FAILED → FAILED: 3回リトライ後も失敗

【全遷移パターン一覧】
UPLOADED    → QUEUED        : キューpush成功時
QUEUED      → PROCESSING    : Workerジョブ取得時
PROCESSING  → COMPLETED     : AI処理成功 + 信頼度 ≥ 閾値
PROCESSING  → NEEDS_REVIEW  : AI処理成功 + 信頼度 < 閾値
PROCESSING  → FAILED        : AIエラー / タイムアウト (3回失敗)
NEEDS_REVIEW → COMPLETED    : User承認
NEEDS_REVIEW → FAILED       : User却下
FAILED      → QUEUED        : 手動リトライ (最大3回まで)
```

## Slide 11: 機能詳細 — F025 一括アップロード

```
機能ID:  F025
機能名:  一括アップロード（バッチアップロード）
優先度:  必須
対象:    ログイン済み Accountant / Manager / Admin

【概要】
最大50ファイル（画像/PDF）を一度に選択し、S3にアップロード後、
Redisキューに複数のAI処理ジョブを積む。ファイルごとに独立して
処理され、1ファイルの失敗が他に影響しない設計とする。

【前提条件】
- ユーザーがログイン済みであること
- アップロードするファイルが手元に存在すること
- ブラウザまたはFlutterアプリを使用していること

【入力項目・制約】
| フィールド        | 必須 | 形式・制約                        |
|-----------------|------|----------------------------------|
| ファイル（複数） | ✓    | jpg/jpeg/png/pdf、各10MB以内      |
| ファイル数       | ✓    | 最小1件、最大50件                  |
| 書類種別 (任意)  | ❌   | 選択時はAIの初期分類に影響         |
| 処理メモ (任意)  | ❌   | テキスト、最大200文字              |

【基本フロー】
1. ユーザーが「一括アップロード」画面を開く
2. ユーザーがファイルを複数選択（ドラッグ＆ドロップ or ファイル選択ダイアログ）
3. システムがクライアントサイドでファイルを検証（件数・種別・サイズ）
4. プレビューリストを表示（ファイル名・サイズ・種別アイコン）
5. ユーザーが「アップロード開始」をクリック
6. システムがBatch レコードを作成（status=BATCH_CREATED）
7. システムがファイルを順次S3へアップロード（プログレスバー表示）
   a. ファイルごとに invoice レコードを作成（status=UPLOADED）
   b. アップロード進捗をリアルタイム更新（例: 3/50完了）
8. 全ファイルのS3アップロード完了後、Redis キューにジョブを一括プッシュ
   a. 各インボイスのステータスを QUEUED に更新
9. ユーザーに「バッチ処理を開始しました」を表示（F034 処理状況へのリンク付き）
10. バックグラウンドでWorkerがジョブを処理
11. 全ジョブ完了後、システムがバッチ完了サマリー通知を送信（F071）

【代替フロー】
3a. ファイルが50件を超える場合
→ クライアントサイドで即座にエラー表示
→ 「一括アップロードは最大50ファイルです。〇〇ファイルを超えています」
→ アップロードは開始しない

3b. 不対応ファイル形式（例: .docx）が含まれる場合
→ 該当ファイルを赤くハイライト表示
→ 「対応形式: JPG, PNG, PDF のみです」
→ 問題ファイルを除外して残りのみでアップロード継続するか確認する

3c. 個別ファイルが10MBを超える場合
→ 超過ファイルを赤くハイライト
→ 「ファイル名.pdf は10MBを超えています（実サイズ: X.X MB）」
→ 問題ファイルを除外して残りのみでアップロード継続するか確認する

7a. 個別ファイルのS3アップロード失敗（ネットワークエラー等）
→ 該当ファイルの invoice ステータスを FAILED に設定
→ エラー理由をログに記録
→ 残りのファイルのアップロードは継続
→ 画面に「〇件のアップロードに失敗しました。詳細を確認してください」

【例外フロー】
EX-1: バッチ内の失敗率が50%超の場合（FAILED_CRITICAL）
→ バッチ全体のステータスを BATCH_FAILED_CRITICAL に設定
→ Manager / Admin にエラーアラート通知（F073）を送信
→ 成功したファイルはそのまま処理継続
→ 失敗ファイルは F032 から個別リトライ可能

EX-2: S3が完全に使用不可の場合
→ バッチ処理を即座に中断
→ バッチステータスを BATCH_FAILED に設定
→ エラーメッセージ [ERR-UPLOAD-002] を表示
→ 「ストレージサービスに接続できません。しばらく待って再試行してください」
→ システム管理者にアラートを送信

EX-3: Redisキューがダウンの場合
→ S3アップロードは完了しているが、ジョブを積めない
→ バッチステータスを BATCH_QUEUED_FAILED に設定
→ エラーメッセージ [ERR-QUEUE-001] を表示
→ Admin画面から手動でキューを再投入できること
→ アップロード済みファイルは削除しない

【バリデーション】
| 項目 | チェック | エラー時 |
|------|---------|---------|
| ファイル数 | 1〜50件 | ERR-UPLOAD-001 |
| ファイル形式 | jpg/jpeg/png/pdf | ERR-UPLOAD-003 |
| ファイルサイズ | 各10MB以内 | ERR-UPLOAD-004 |
| ファイル名 | 重複ファイル名を警告 | 警告のみ（続行可） |
| バックエンド | ファイル形式はサーバー側でも再検証 | ERR-UPLOAD-003 |
| バックエンド | JWT有効期限確認 | ERR-AUTH-001 → ログイン画面へ |

【エラーコード一覧】
| コード | 内容 | 対処 |
|--------|------|------|
| ERR-UPLOAD-001 | ファイル数超過 (>50) | ファイルを減らして再試行 |
| ERR-UPLOAD-002 | S3接続エラー | 管理者に連絡 |
| ERR-UPLOAD-003 | 非対応ファイル形式 | JPG/PNG/PDFに変換 |
| ERR-UPLOAD-004 | ファイルサイズ超過 | 10MB以下に圧縮 |
| ERR-QUEUE-001 | Redisキュー接続エラー | 管理者に連絡 |
| ERR-AUTH-001 | 認証エラー（JWT期限切れ） | 再ログイン |

【バッチステータス定義】
| ステータス | 説明 | 次に遷移可能 |
|-----------|------|------------|
| BATCH_CREATED | バッチレコード作成完了 | BATCH_UPLOADING |
| BATCH_UPLOADING | S3への順次アップロード中 | BATCH_QUEUED, BATCH_FAILED |
| BATCH_QUEUED | 全ジョブをRedisキューに投入済み | BATCH_PROCESSING |
| BATCH_PROCESSING | Worker処理中 | BATCH_COMPLETED, BATCH_FAILED_CRITICAL |
| BATCH_COMPLETED | 全ジョブ完了（一部FAILEDあり可） | なし |
| BATCH_FAILED | S3またはキューの全体エラー | なし（手動リトライ可） |
| BATCH_FAILED_CRITICAL | 50%超のファイルがFAILED | なし（手動リトライ可） |

【関連機能】
- F031: 処理状況リアルタイム更新（バッチ進捗表示に使用）
- F032: 失敗リトライ（FAILED個票を再処理）
- F034: バッチ処理状況画面
- F071: バッチ完了サマリー通知
- F073: エラーアラート通知
```

## Slide 13: まとめと宿題

### まとめ
- AI処理は**非同期** — UPLOADEDからCOMPLETEDまで全Stateを定義しないと設計が崩れる
- バッチ処理は「1件失敗 = 全件失敗」ではなく「**個別失敗・継続処理**」が基本設計
- セキュリティは設計段階で組み込む — API key はフロントに出さない、S3は signed URL
- 月間8,000件のデータ = 検索・フィルタ・ページネーション設計が必須

### 宿題
> 以下の機能詳細を完成させること（次回レビューあり）:
>
> **F040 AI結果レビュー** — 分割画面での確認フロー、信頼度表示、承認・却下の全State遷移
>
> **F072 AI処理失敗通知** — FAILEDになった時の通知先・通知内容・リトライへの導線
>
> **ヒント (F040):**
> - 前提条件: invoice.status = NEEDS_REVIEW であること
> - 画面左: S3のSigned URLで証憑画像を表示（期限付きURL）
> - 画面右: AI抽出フィールド + 信頼度% + 勘定科目提案
> - 承認時: journal_entry レコードを作成 → status = COMPLETED
> - 却下時: 理由必須 → status = FAILED
