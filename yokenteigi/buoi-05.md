## Slide 3: Chuẩn bị trước buổi ヒアリング

### Checklist chuẩn bị (Meeting Prep)

**1. Nghiên cứu trước**
- [ ] Đọc tài liệu khách hàng đã cung cấp
- [ ] Tìm hiểu về ngành nghề, quy mô công ty
- [ ] Nghiên cứu hệ thống hiện tại (nếu có)

**2. Chuẩn bị câu hỏi**
- [ ] Danh sách câu hỏi theo 5W1H
- [ ] Ưu tiên câu hỏi theo mức độ quan trọng
- [ ] Dự phòng câu hỏi follow-up

**3. Logistics**
- [ ] Gửi agenda trước ít nhất 2 ngày
- [ ] Xác nhận participants (người ra quyết định có tham gia không?)
- [ ] Chuẩn bị template 議事録
- [ ] Sắp xếp người ghi chép (別に議事録担当)

## Slide 4: Cấu trúc buổi ヒアリング hiệu quả

### Format chuẩn 90 phút

```
┌─────────────────────────────────────────────┐
│ 0:00 - 0:05  自己紹介・アイスブレイク         │
│ 0:05 - 0:10  アジェンダ確認                  │
│ 0:10 - 0:40  現状の業務・課題のヒアリング     │
│              (As-Is理解)                     │
│ 0:40 - 1:05  新システムへの要望ヒアリング     │
│              (To-Be期待)                     │
│ 1:05 - 1:20  優先順位・制約の確認             │
│ 1:20 - 1:25  次回アクション確認              │
│ 1:25 - 1:30  クロージング                   │
└─────────────────────────────────────────────┘
```

### Nguyên tắc vàng
> Khách hàng nói **70%**, bạn nói **30%**
> Câu hỏi mở trước, câu hỏi đóng để confirm sau

## Slide 5: Kỹ thuật 5W1H trong ヒアリング

### Framework hỏi chuẩn

| W/H | Câu hỏi mẫu | Mục đích |
|-----|------------|---------|
| **What** | 「現在どのような業務を行っていますか？」 | Hiểu scope |
| **Why** | 「なぜその処理が必要ですか？」 | Hiểu động cơ |
| **Who** | 「誰がこの機能を使いますか？」 | Xác định actor |
| **When** | 「いつ処理が発生しますか？頻度は？」 | Timing, frequency |
| **Where** | 「どのような環境で使いますか？」 | Context |
| **How** | 「現在どのような手順で行っていますか？」 | Quy trình hiện tại |

### Kỹ thuật "なぜ×5" (5 Whys)

```
Khách: "注文を手動で入力したい"
BA:    "なぜ手動で入力するのですか？"
Khách: "電話で受けた注文があるので"
BA:    "なぜ電話注文が必要ですか？"
Khách: "高齢のお客様がネットを使えないので"
BA:    "なぜそのお客様は電話しか使えないのですか？"
Khách: "主に60代以上の固定客です"
BA:    "電話注文は全体の何%くらいですか？"
Khách: "約20%です"

→ 発見: 電話注文専用の管理者画面が必要
  (最初のヒアリングでは出てこなかった要件)
```

## Slide 6: Câu hỏi mở vs câu hỏi đóng

### Câu hỏi mở (Open Questions) — Dùng để khám phá

**Ví dụ:**
- 「現在の業務で一番困っていることは何ですか？」
- 「新しいシステムに期待することを教えてください」
- 「どのような場合にエラーが起きますか？」

### Câu hỏi đóng (Closed Questions) — Dùng để confirm

**Ví dụ:**
- 「同時に使用するユーザーは最大100人でよろしいですか？」
- 「データの保持期間は5年でよろしいでしょうか？」
- 「モバイル対応は今回のスコープ外でよろしいですか？」

### Pattern confirm chuẩn Nhật
```
「〇〇という理解でよろしいでしょうか？」
(= "Tôi hiểu là X, có đúng không ạ?")

「確認なのですが、△△の場合は□□という認識でよいですか？」
(= "Cho tôi xác nhận, trong trường hợp X thì Y đúng không?")
```

## Slide 7: Xử lý các tình huống khó

### Tình huống 1: Khách hàng mơ hồ

**KH nói:** "なんとなくいい感じにしてほしい" (Làm cho nó tốt hơn)

**Cách xử lý:**
```
「"いい感じ"とは具体的にどういう状態ですか？
 例えば、現在は〇〇という問題がありますが、
 それが解決されれば"いい感じ"と言えますか？」
```
→ Từ mơ hồ → định nghĩa cụ thể bằng cách đề xuất tiêu chí

---

### Tình huống 2: Yêu cầu mâu thuẫn

**Ví dụ:**
- Phòng A: "全データを1年間保持してください"
- Phòng B: "コストを下げるためデータは3ヶ月で削除してください"

**Cách xử lý:**
```
1. Ghi nhận cả hai yêu cầu vào 課題管理表
2. Escalate lên người ra quyết định (stakeholder cao hơn)
3. Không tự ý quyết định
4. Đặt deadline để giải quyết
```

---

### Tình huống 3: Yêu cầu bất khả thi

**KH:** "全機能を1ヶ月で完成させてください"

**Cách xử lý:**
```
「ご要望の機能を精査した結果、品質を確保するには
 最低でも3ヶ月必要と見込んでいます。

 代替案として:
 ①フェーズ1（必須機能のみ）：1.5ヶ月
 ②フェーズ2（残り機能）：1.5ヶ月
 という進め方はいかがでしょうか？」
```
→ 問題提起 + 代替案セット で提示する

## Slide 10: Checklist 30 điểm tự review tài liệu Yokenteigi

### Nhóm 1: 文書基本 (5 điểm)
- [ ] 改訂履歴が最新か
- [ ] バージョン番号が正しく付与されているか
- [ ] 用語定義が一貫しているか
- [ ] 参考文書が列挙されているか
- [ ] 承認欄があるか

### Nhóm 2: システム概要 (3 điểm)
- [ ] 目的と背景が明確か
- [ ] ステークホルダーが全員列挙されているか
- [ ] システム構成図があるか

### Nhóm 3: 業務フロー (4 điểm)
- [ ] As-IsとTo-Beの両方があるか
- [ ] 全ステークホルダーのSwimlaneがあるか
- [ ] 例外フロー（エラー、在庫なしなど）が描かれているか
- [ ] 現状の課題が明記されているか

### Nhóm 4: 機能要件 (8 điểm)
- [ ] 機能一覧に全機能が網羅されているか
- [ ] 機能IDが重複・漏れなく付与されているか
- [ ] 優先度が全機能に設定されているか
- [ ] 基本フローが全機能にあるか
- [ ] 代替フローが必要な機能にあるか
- [ ] 例外フローが全機能にあるか
- [ ] バリデーション仕様が定義されているか
- [ ] エラーメッセージが定義されているか

### Nhóm 5: 非機能要件 (5 điểm)
- [ ] 性能要件に具体的な数値があるか
- [ ] 可用性（稼働率、RTO/RPO）が定義されているか
- [ ] セキュリティ要件が最低5項目あるか
- [ ] バックアップ要件が定義されているか
- [ ] 測定条件（何人同時などの前提）が明記されているか

### Nhóm 6: その他 (5 điểm)
- [ ] 画面遷移図が全画面をカバーしているか
- [ ] データ要件に保持期間があるか
- [ ] 外部IF要件に全連携先があるか
- [ ] 制約・前提条件が明記されているか
- [ ] 課題管理表に担当者と期限があるか

## Slide 11: Role-Play ヒアリング実習 (30分)

### セットアップ

**シナリオ:** 中堅会計事務所「AIチャレンジ会計事務所」の業務自動化システム化

**役割分担:**
- 2名: 顧客役（カード配布）
- 2名: BA役（ヒアリング担当）
- 残り: オブザーバー（評価シート記入）

**顧客カード（BA役は見ない）:**
```
あなたは「AIチャレンジ会計事務所」の所長です。
・スタッフ30名で1日500件の請求書を手入力処理している
・入力ミスが月50件発生しており、クライアントからクレームが来ている
・スタッフの残業が平均2時間/日に達しており、離職率が上がっている

・現在「弥生会計」を使っており、新システムとの連携が必要

```

**評価ポイント:**
- 5W1Hで質問できているか
- 数値を引き出せているか（件数、ミス件数、残業時間など）
- Pain Pointを特定できているか（精度への不安、弥生会計連携など）
- 議事録を正確に取れているか

## Slide 12: まとめと宿題

### まとめ
- ヒアリングの成功 = 事前準備 × 質問技術 × 議事録
- Open Question で探索 → Closed Question で確認
- 議事録は24時間以内、決定事項・未決事項を分離
- 質問書で複雑な質問を文書化 → 証拠として残す

### 宿題
> ロールプレイで出てきた情報を元に:
> 1. **議事録**を正式フォーマットで作成する（参加者、決定事項、未決事項、次回アクションを含めること）
> 2. **質問書**フォーマットにまとめる（最低5件） — 必ず含めること: (a) AIの認識精度の許容ラインは何%か、(b) 1日の請求書処理件数のピーク値、(c) 勘定科目マスタの件数と管理方法、(d) 弥生会計との連携方式（API/CSV等）、(e) 既存データの移行要否
> 3. 以下の機能について**機能詳細**を書く:
>    「**AI結果の一括承認機能**」 — 複数の請求書（NEEDS_REVIEWステータス）をまとめて一括承認する機能。前提条件、基本フロー（件数上限は？承認後のステータス変化は？）、例外フロー（一部だけ承認できない場合は？）を含めること
