### Lỗi phổ biến: F034 申請承認

**❌ Thiếu: Không xử lý trạng thái thiết bị**
```
【基本フロー】
1. 申請一覧を確認
2. 「承認」ボタンをクリック
3. 申請ステータスを「承認済み」に変更
```

**✅ Đầy đủ: Bao gồm cả state transition**
```
【基本フロー】
1. Adminが申請一覧（F033）を確認
2. 申請詳細を開く（申請者・機器・期間・目的）
3. Adminが「承認」ボタンをクリック
4. システムが確認ダイアログを表示
5. Adminが「確定」をクリック
6. システムが申請ステータスを「申請中」→「承認済み」に変更
7. システムが機器ステータスを「予約済み」→「貸出中」に変更
   ※ 貸出開始日が未来の場合は「予約済み」のまま維持
8. システムがUserに承認通知メールを送信（F051）
9. 申請一覧画面に戻り、「承認しました」を表示

【ポイント】
- 貸出開始日が「本日」の場合 → 即「貸出中」
- 貸出開始日が「未来」の場合 → 「予約済み」のまま
  → バッチが開始日当日に「貸出中」へ自動更新
```

### Lỗi phổ biến: F053 返却期限超過通知

**❌ Thiếu: Không định nghĩa điều kiện batch**
```
返却期限を過ぎたら通知を送る
```

**✅ Đầy đủ:**
```
【バッチ実行仕様】
実行タイミング: 毎日 08:00
処理内容: 返却予定日が本日以前で「貸出中」の全申請を抽出

【通知ルール】
| 超過日数 | Userへの通知 | Adminへの通知 |
|---------|-------------|--------------|
| 超過当日 | メール送信（1回目） | メール送信 |
| 超過3日後 | メール送信（2回目） | メール送信 |
| 超過7日後 | メール送信（3回目・最終） | メール + 要対応アラート |
| 超過14日以降 | システム画面に警告バナー | エスカレーション通知 |

【メール本文の必須要素】
- 機器名・機器ID
- 貸出開始日・返却期限日・超過日数
- 返却申請ページへのリンク
- 担当者連絡先

【例外フロー】
- 送信失敗: ログに記録、翌バッチで再送（最大3回）
- 既に返却処理済みの場合: スキップ（重複送信しない）
```

## Slide 3: 機能詳細 — F041 返却確認 (モデル解答)

```
機能ID:  F041
機能名:  返却確認
優先度:  必須
対象:    Admin

【概要】
社員が物理的に機器を返却した後、Adminがシステムで
返却確認処理を行い、機器ステータスを「利用可能」に戻す。

【前提条件】
- 対象申請のステータスが「貸出中」であること
- 社員が物理的に機器を返却済みであること（システム外確認）

【基本フロー】
1. Adminが申請一覧（F033）で「貸出中」の申請を確認
2. 返却された機器の申請詳細を開く
3. Adminが機器の外観・動作確認を行う（システム外）
4. Adminが「返却確認」ボタンをクリック
5. システムが返却確認ダイアログを表示
   - 機器名、申請番号、返却日（本日）を表示
   - 「異常あり」チェックボックス（任意）
6. Adminが確認して「完了」をクリック
7. システムが処理を実行:
   a. 申請ステータス: 「貸出中」→「返却済み」
   b. 機器ステータス: 「貸出中」→「利用可能」
   c. 実際の返却日時を記録
8. システムがUserに返却完了通知を送信（F054）
9. 申請一覧に戻り「返却確認が完了しました」を表示

【代替フロー】
6a. 「異常あり」にチェックした場合
→ 機器ステータスを「メンテナンス中」に変更（「利用可能」にしない）
→ 理由入力フィールドを表示（必須、最大500文字）
→ Adminに「機器異常報告」メールを別途送信
→ 申請は「返却済み（要確認）」ステータスに変更

【例外フロー】
7a. DB更新エラー
→ トランザクションロールバック（全ての変更を取り消す）
→ エラーメッセージ[ERR-RETURN-001]を表示
→ 申請・機器ステータスは変更前のままを維持

8a. 通知メール送信失敗
→ 申請・機器ステータスは更新済みを維持
→ 画面に「Userへの通知に失敗しました」を表示
→ 15分後に1回自動再送
```

## Slide 4: 非機能要件 — 機器管理システム向け

### このシステムの特性

| 特性 | 影響する非機能要件 |
|------|----------------|
| 社内システム（社員のみ利用） | 性能要件は高くなくてよい |

### 5.3 セキュリティ要件

```
5.3 セキュリティ要件

【認証・認可】
- パスワード: 8文字以上、英数字混在
- ハッシュ: bcrypt（コストファクター12）
- ログイン失敗: 5回連続でアカウントロック（Admin解除）
- セッションタイムアウト: 無操作60分（業務利用のため長め）

【アクセス制御】
- UserはAdmin機能のURLに直接アクセスしても403エラー
- 他人の申請詳細へのURL直打ちアクセスを禁止
- 全てのAPI: 認証トークン必須

【通信セキュリティ】
- 全通信: HTTPS（TLS 1.2以上）
- 社内ネットワーク or VPN接続時のみアクセス可能
  ※ 社外（在宅勤務）からはVPN経由でアクセス

【監査ログ（必須）】
以下の操作を全て記録する:
- ログイン・ログアウト（成功・失敗）
- 機器のCRUD操作（誰が・いつ・何を変更したか）
- 申請・承認・却下・返却の全操作

【データ保護】
- 機器情報は社内資産情報のため、外部送信禁止
- CSVエクスポートはAdmin権限のみ、操作ログに記録
```

### 9. 制約・前提条件

**制約条件**
- 利用環境: 社内ネットワーク or VPN必須（外部公開しない）
- DB: PostgreSQL（既存インフラに合わせる）
- 開発期間: 3ヶ月（スプリント3回）
- 予算: 開発費200万円以内、運用費5万円/月以内

**前提条件**
- 初期データ: 現在のExcel（約500件）をCSVインポートで移行
- 移行データの整合性確認は別途移行計画書で定義
- スマートフォン対応: レスポンシブデザインで対応（専用アプリは作らない）
- AD連携: 本プロジェクトでは独自ログインを使用、Phase 2でLDAP連携
