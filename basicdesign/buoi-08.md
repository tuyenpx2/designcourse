## Slide 4: よくある失敗 — 実務から10の教訓

### 教訓 1: DBを変えると全部変わる
> リリース後にDBカラムを追加するのは想像以上に大変。
> スキーマレビューは「3回読み直す」くらい慎重に。

---

### 教訓 2: PaginationなしのList APIは時限爆弾
> 最初は100件で動く。1年後は10万件でタイムアウト。
> 全てのList APIには最初からPaginationを実装する。

---

### 教訓 3: ソフトウェアの設計書はコードと乖離する
> 設計書を更新しないと「嘘のドキュメント」になる。
> コードを変えたら設計書も同時に更新するルールを作る。

---

### 教訓 4: 楽観ロックを忘れると本番障害
> 「同時アクセスなんてない」は大間違い。
> 特に申請・承認ワークフローには必須。

---

### 教訓 5: エラーコードは最初から決める
> 後から追加するとフロントエンドとの認識がズレる。
> `ERR-MODULE-CODE` 形式で最初に全部定義する。

---

### 教訓 6: Audit Logは「後から追加」が最も難しい
> 本番データが汚染された時、追う手段がない。
> セキュリティ要件の前に「何を記録するか」を設計に入れる。

---

### 教訓 7: セキュリティは後付けできない
> localStorageにJWTを保存 → XSSで全トークン盗まれる。
> 設計段階で「httpOnly Cookie + CSRF対策」を決定する。

---

### 教訓 8: Batchの冪等性は最初から保証する
> 二重実行でメールが2通届く → ユーザークレーム。
> 「同じ条件で2回実行しても同じ結果」を設計原則とする。

---

### 教訓 9: マイグレーションは必ず順番通りに
> V005を先に実行してV003をスキップ → FK制約エラー。
> 番号とdependencyを設計書に明記する。

---

### 教訓 10: レビューなしで詳細設計に進まない
> 基本設計のGapは詳細設計で「倍になって返ってくる」。
> KH + BA + Leadによる正式レビューと承認を必ず取る。

## Slide 5: 実務での設計書レビュー視点

### あなたがレビュアーになったとき、まず見るべき5点

**1. ER図を10秒で見て「それっぽいか」を判断**
> N:Mのままになっていないかをまずチェック。
> 仲介テーブルがないN:Mはほぼ設計ミス。

**2. 全テーブルのPK・created_at・updated_atを確認**
> 2分でわかるチェック。基本が抜けていたら深掘り不要、やり直し指示。

**3. 一番重要なList APIのレスポンス定義を見る**
> Paginationがあるか、JOINの設計が効率的かを確認。
> N+1問題を誘発しそうなら指摘。

**4. エラーコード一覧が存在するか**
> なければ即アウト。あればランダムな番号がないかチェック。

**5. セキュリティ設計に認可チェックが書いてあるか**
> 「認証済みなら全てアクセス可」という設計は危険。
> Admin専用の操作に権限チェックがあるか確認。

## Slide 6: 基本設計から詳細設計へ — 次のステップ

### 詳細設計で追加で決めること

| 項目 | 基本設計 | 詳細設計 |
|------|---------|---------|
| DB | テーブル定義・FK・Index | SQL Query・Explain分析 |
| API | endpoint・request/response | 処理ロジックのフローチャート |
| 画面 | Wireframe・データソース | HTML構造・CSS設計・エラーメッセージ文言 |
| Batch | スケジュール・フロー | 実際のSQL・並行処理設計 |
| Security | 方針・実装方式 | 具体的なライブラリ・設定値 |

### 詳細設計書の典型的な目次

```
詳細設計書
│
├── 1. 機能詳細設計
│   └── 各機能の処理フローチャート + 擬似コード
│
├── 2. 画面詳細設計
│   └── 各画面の項目定義・バリデーション詳細・HTMLモックアップ
│
├── 3. DB詳細設計
│   └── 全SQLクエリ・EXPLAINプラン・マイグレーション詳細
│
├── 4. API詳細設計
│   └── OpenAPI (Swagger) 仕様書完全版
│
└── 5. テスト設計
    └── 単体テスト・結合テストのシナリオ
```

## Slide 7: 設計スキルのキャリアへの活かし方

### ポジション別の活用

**SE として:**
```
基本設計書を書ける = 「設計できる SE」
→ チームのリード・メンター役
→ 年収帯: ワンランク上
```

**BA として:**
```
設計を理解している = 「技術的に話せる BA」
→ KH ↔ Dev の間で「設計的に可能か」を判断できる
→ プロジェクトの価値が倍増
```

**PM として:**
```
設計を読める = 「スコープを正確に見積もれる PM」
→ 「これは1日で終わる？1週間？」の精度が上がる
→ リスク管理・変更管理の精度向上
```

### 次に学ぶべきこと

| スキル | 理由 |
|--------|------|
| 詳細設計 (Shousai Sekkei) | 基本設計の次のフェーズ |
| OpenAPI (Swagger) | API仕様書の業界標準ツール |
| データモデリング (正規化理論) | DB設計の理論的基盤 |
| クラウドアーキテクチャ (AWS/GCP) | インフラ設計の現代的スキル |
| セキュリティ (OWASP) | 全ての設計者に必須 |

## Slide 8: 設計書の黄金ルール 12箇条

```
①  DB設計はプロジェクト最初に決める
    → スキーマ変更のコストは時間とともに指数関数的に増加

②  N:Mは必ず仲介テーブルに落とす
    → 配列型カラムに逃げない

③  Soft Deleteの対象テーブルを最初に決める
    → 後から追加すると既存データが壊れる

④  楽観ロックを競合しうる全リソースに適用する
    → version カラムは最初から全テーブルに入れてもよい

⑤  Error Codeは設計書で一元管理する
    → コードベースに散らばったエラーハンドリングを防ぐ

⑥  全List APIにPaginationを実装する
    → 例外なし

⑦  JWT is stateless — 機密情報を入れない
    → httpOnly CookieでXSSから守る

⑧  Batch Jobは必ず冪等性を保証する
    → 「2回実行で2回処理」は禁止

⑨  トレーサビリティマトリックスを常に更新する
    → 要件と設計のGapを「見える化」する

⑩  セキュリティ設計は「後から追加できない」
    → 認証・認可・XSS・SQLiは設計フェーズで決める

⑪  設計書はコードと同期させる
    → コードが変わったら設計書も変える

⑫  レビューなしに次フェーズに進まない
    → 基本設計のサインオフは必須
```
