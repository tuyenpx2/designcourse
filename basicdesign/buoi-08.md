## Slide 4: よくある失敗 — 実務から10の教訓

### 教訓 1: DBを変えると全部変わる
> リリース後にDBカラムを追加するのは想像以上に大変。
> `invoices.confidence_threshold` を後から追加 → invoice_extracted_data との整合性確認が地獄。
> スキーマレビューは「3回読み直す」くらい慎重に。

---

### 教訓 2: PaginationなしのList APIは時限爆弾
> 最初は100件で動く。1年後は10,000件でタイムアウト。
> `/journal-entries` は5年で7万件超える。全てのList APIには最初からPaginationを実装する。

---

### 教訓 3: ソフトウェアの設計書はコードと乖離する
> 設計書を更新しないと「嘘のドキュメント」になる。
> コードを変えたら設計書も同時に更新するルールを作る。

---

### 教訓 4: 非同期処理の状態設計を甘く見ると本番障害
> 「PROCESSINGのまま止まった請求書」は実際のシステムで必ず発生する。
> BATCH-001（待機中請求書の再キュー）がないと手動対応が発生し続ける。

---

### 教訓 5: エラーコードは最初から決める
> 後から追加するとフロントエンドとの認識がズレる。
> `ERR-MODULE-CODE` 形式で最初に全部定義する。
> 特に `ERR-AI-001` (AI Service down) は本番で必ず起きる。

---

### 教訓 6: Audit Logは「後から追加」が最も難しい
> 財務データが改ざんされた時、追う手段がない。
> AI-IAでは会計帳簿の証跡として7年保持義務がある。
> 「何を記録するか」を設計に最初から入れる。

---

### 教訓 7: AI API Keyを絶対にフロントエンドに渡さない
> Flutter アプリはリバースエンジニアリングされる。
> AI Service の API Key がバレると → 攻撃者がAI処理を無制限に実行できる。
> 設計段階で「Python AI Service は Laravel からのみ呼び出す」を決定する。

---

### 教訓 8: Batchの冪等性は最初から保証する
> BATCH-002 が2回実行されると同じ請求書が2回リトライされる。
> `AND status = 'FAILED' AND retry_count < 3` の条件でCAS更新を設計原則とする。

---

### 教訓 9: マイグレーションは必ず順番通りに
> V006 (invoice_extracted_data) を V005 (invoices) より先に実行 → FK制約エラー。
> 番号と dependency を設計書に明記する。

---

### 教訓 10: レビューなしで詳細設計に進まない
> 基本設計のGapは詳細設計で「倍になって返ってくる」。
> KH + BA + Leadによる正式レビューと承認を必ず取る。

## Slide 5: 実務での設計書レビュー視点

### あなたがレビュアーになったとき、まず見るべき5点

**1. AI Processing State Machineが完全か**
> 6ステータス全て (UPLOADED/QUEUED/PROCESSING/COMPLETED/FAILED/NEEDS_REVIEW) が設計に反映されているか。
> FAILEDからのリトライ設計があるか。PROCESSINGタイムアウトの設計があるか。

**2. Internal APIが適切に保護されているか**
> POST /ai/process が Nginx に expose されていないか。
> AI Service API Key の管理方針が明記されているか。
> Flutter からAI Serviceに直接アクセスできない設計になっているか。

**3. 一番重要なList APIのレスポンス定義を見る**
> Paginationがあるか。100,000件超でのパフォーマンスを考慮したIndexがあるか。
> `(status, created_at)` 複合Indexが invoices テーブルにあるか。

**4. ファイルセキュリティの設計**
> MinIO Signed URL の有効期限が定義されているか (30分)。
> ファイルアップロードのバリデーション (MIME+マジックバイト+ClamAV) が設計されているか。
> `invoices.file_path` が公開URLではなくMinIOオブジェクトキーになっているか。

**5. 財務データの完全性**
> `amount`, `total_amount` が NUMERIC (FLOATではない) か。
> `confidence_score` が NUMERIC(5,4) か。
> `journal_entries` に物理削除禁止の注記があるか。

## Slide 6: 基本設計から詳細設計へ — 次のステップ

### 詳細設計で追加で決めること

| 項目 | 基本設計 | 詳細設計 |
|------|---------|---------|
| DB | テーブル定義・FK・Index | SQL Query・EXPLAIN分析・パーティション設計 |
| API | endpoint・request/response | 処理ロジックのフローチャート・Laravel Controller詳細 |
| 画面 | Wireframe・データソース | Flutter Widget設計・Riverpod状態管理 |
| AI連携 | State Machine・タイムアウト | Laravel Queue Job実装・Python FastAPI endpoint |
| Batch | スケジュール・フロー | 実際のSQL・Laravel Artisan Command実装 |
| Security | 方針・実装方式 | Laravel Middleware・ClamAV統合コード |

### 詳細設計書の典型的な目次

```
詳細設計書
│
├── 1. 機能詳細設計
│   └── 各機能の処理フローチャート + 擬似コード
│   └── AI処理Worker の詳細フロー
│
├── 2. 画面詳細設計
│   └── Flutter Widget 構成・状態管理 (Riverpod)
│   └── Pollingロジックの実装仕様
│
├── 3. DB詳細設計
│   └── 全SQLクエリ・EXPLAINプラン
│   └── パーティション設計 (大量データ時)
│
├── 4. API詳細設計
│   └── OpenAPI (Swagger) 仕様書完全版
│   └── Internal API (Laravel↔Python) 詳細
│
└── 5. テスト設計
    └── 単体テスト・結合テスト・AI精度テスト
```

## Slide 7: 設計スキルのキャリアへの活かし方

### ポジション別の活用

**SE として:**
```
基本設計書を書ける = 「設計できる SE」
→ AI/ML システムの設計経験 → 付加価値
→ 非同期処理・マイクロサービス経験 → モダンシステムに対応
→ 年収帯: ワンランク上
```

**BA として:**
```
設計を理解している = 「技術的に話せる BA」
→ 「AI の confidence_score しきい値は業務要件で決まる」を判断できる
→ KH の「AIの精度が低い」という要件を「0.85 → 0.90 に変更」に変換できる
```

**PM として:**
```
設計を読める = 「スコープを正確に見積もれる PM」
→ 「非同期処理の追加は State Machine の変更が必要」と判断できる
→ 変更管理の精度向上
```

### 次に学ぶべきこと

| スキル | 理由 |
|--------|------|
| 詳細設計 (Shousai Sekkei) | 基本設計の次のフェーズ |
| OpenAPI (Swagger) | API仕様書の業界標準ツール |
| データモデリング (正規化理論) | DB設計の理論的基盤 |
| クラウドアーキテクチャ (AWS/GCP) | AI/MLシステムのインフラ設計 |
| セキュリティ (OWASP) | 全ての設計者に必須 |
| AI/ML システム設計 | AI-IAで学んだスキルの発展 |

## Slide 8: 設計書の黄金ルール 12箇条

```
①  DB設計はプロジェクト最初に決める
    → スキーマ変更のコストは時間とともに指数関数的に増加
    → AI-IAの場合: invoices.status の6値は最初に確定させる

②  N:Mは必ず仲介テーブルに落とす
    → 配列型カラムに逃げない

③  Soft Deleteの対象テーブルを最初に決める
    → 特に財務データは法的に保持義務がある

④  非同期処理の「ターミナル状態」を明確に定義する
    → AI-IAの場合: COMPLETED/FAILED/NEEDS_REVIEW が is_terminal = true
    → Flutterがポーリングを停止するタイミングを設計書に明記

⑤  Error Codeは設計書で一元管理する
    → ERR-AI-001 (AIサービス障害) は特に早期に定義する

⑥  全List APIにPaginationを実装する
    → 例外なし。journal_entries は5年後に7万件超える。

⑦  Internal APIは絶対に外部に公開しない
    → POST /ai/process は Laravel から Python へのサーバー間通信のみ
    → AI API Key は .env のみ、Flutterには絶対に渡さない

⑧  Batch Jobは必ず冪等性を保証する
    → 「2回実行で2回リトライ」は禁止
    → CAS更新 (AND status = '期待値') で排他制御

⑨  トレーサビリティマトリックスを常に更新する
    → 要件と設計のGapを「見える化」する

⑩  セキュリティ設計は「後から追加できない」
    → 認証・認可・ファイルアップロード制限・Signed URLは設計フェーズで決める

⑪  設計書はコードと同期させる
    → コードが変わったら設計書も変える
    → confidence_score の閾値変更は設計書にも反映

⑫  レビューなしに次フェーズに進まない
    → 基本設計のサインオフは必須
    → AI-IAの場合: AI State Machine のレビューを特に念入りに
```

